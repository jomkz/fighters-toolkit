## Requires Ghidra GUI (FA.EXE project)

The remaining items genuinely require the interactive GUI. Three distinct blockers apply across these items — none can be resolved by improving the headless scripts:

1. **Pointer/vtable indirection**: code accesses fields via `*(ptr + offset)`, so the target address never appears as a named symbol in `DumpAllFunctions.txt`. "Show References To" xref navigation is the only path.
2. **Overlapping struct layouts**: two structs share the same memory region; static text cannot distinguish which reads belong to which interpretation — requires Ghidra's type-annotation and live re-decompile.
3. **Broken calling convention**: Ghidra headless misidentifies the return type or register-passed argument of a function; the data-flow between caller and callee is severed in the dump and can only be repaired by correcting the function signature in the GUI.

The FA.EXE project is at `%FA_PROJECT%\fa-re.gpr`.

- **AI / BI opcode 0x28 FRAME consumer (reader)** *(blocker: pointer indirection)*: Writer fully confirmed: `FUN_00466a80` case 0x28 (`DumpAllFunctions.txt` line 78118) reads 4 bytes from bytecode stream → `DAT_00546c44`/`DAT_00546c46` (CT state block `+0x7c`/`+0x7e`), advances PC by 4, with a stack-imbalance guard. Reader not found — verified that `DAT_00546c44`/`DAT_00546c46` each appear exactly once in `DumpAllFunctions.txt` (lines 78123–78124), both write-only assignments; no read site exists at any direct address. Access is via a pointer (`*(DAT_0050cf90) + 0x7c`), invisible to static offset scans. All 16 `findFunctionsReadingOffsets` candidates at offset +0x7c were false positives. *Approach: cross-reference `DAT_00546bc8` in the FA.EXE Ghidra GUI to find all struct-pointer loads of the 128-byte CT state block.* See [formats/AI.md](formats/AI.md) and [formats/BI.md](formats/BI.md).

- **JT physics gap (`+0x50`–`+0x54`, residual)** *(blocker: overlapping struct layouts)*: `_PROJMoveProc` (0x4c11b0) decompiled (2026-05-19). Motor phase thresholds, seeker search params, smoke trail, and warhead bits 1/5/6/12/13 confirmed. Remaining: `+0x50`–`+0x54` (entity offset table: reaction params / mode byte) and scattered bytes `+0x56`/`+0x58`/`+0x5A`/`+0x5C`/`+0x5E`–`+0x64` — these addresses overlap the aircraft flight model (BRF entity in scratchpad); no missile-specific reads isolated. `FUN_004c1630` and `FUN_004c1660` were checked and ruled out: `FUN_004c1630` (`AnalyzePROJ.txt` line 104654) is purely a `_CreateMove_52` wrapper with no entity offset reads; `FUN_004c1660` (line 97053) reads `iVar1+0x11`, `+0x15`, `+0x19` — a different range unrelated to the gap. *Approach: Ghidra GUI, annotate the missile JT struct and BRF entity as overlapping types on the scratchpad base pointer, then filter `_PROJMoveProc` for reads at `entity+0xF6`–`0xFA` specifically.* See [formats/JT.md](formats/JT.md).

- **NET `CN_INFO` IPX sub-block (`[0xc0]`–`[0x8e3]`, ~2,180 bytes)** *(blocker: pointer/vtable indirection, cross-binary)*: No direct field accesses for this range visible in the FA.EXE decompile — `_proto_ptr` has zero hits in `AnalyzeNetwork.txt`; all 35 occurrences of `0x3e` there are numeric comparisons (`== 0x3e04` game-state constants), not vtable offset reads. `Overlay_IP.EXE.txt` already exists from headless analysis (58,454 lines, 1,810 functions) but every function is unnamed (`FUN_0040xxxx`); without knowing which slot in the `_proto_ptr` vtable resolves to which IP.EXE function, the sub-block layout cannot be extracted from the dump alone. *Approach: Ghidra GUI xref on `_proto_ptr` in FA.EXE to identify the IPX vtable → follow slot `+0x3e` to its implementation address in IP.EXE → trace field accesses in that function body.* See [formats/NET.md](formats/NET.md).

## Requires Ghidra GUI (Terrain Renderer)

- **T2 surface class → PIC atlas mapping** *(blocker: broken calling convention)*: No terrain overlay DLL exists — `ft lib ls FA_2.LIB` lists no unrecognized types; `FA_1.LIB` contains only FNT/PIC. The BIT2 parser and all terrain rendering functions (`@T_Load@4` 0x4c5d70, `@T_GetLeaf@12` 0x4c6040, `FUN_004d3064` 0x4d3064) are in FA.EXE. `@DoSetTmapRemaps@0` (0x4cc518) builds a 256-byte `_tmapRemapTable` by iterating `Remap()` 256 times; the surface class byte (0–255) is the table index. `Remap()` IS decompiled in the headless output (`DumpAllFunctions.txt` line 160249, address `0x004cc44c`) but the decompile is broken: Ghidra declared its return type as `void` even though the caller assigns `uVar1 = Remap()`, and its sole input is `in_EAX` (register-passed argument not reconstructed from the caller side). The actual data flow — loop counter 0–255 passed in EAX → remap byte returned in EAX → written into `_tmapRemapTable[i]` — is completely severed in the static dump. Sub-header bytes 4–16 class constants and tile-summary record 0 selection algorithm remain open. *Approach: Ghidra GUI, correct `Remap()`'s function signature (set return type to `byte`, mark EAX input), trigger live re-decompile to restore data flow, then trace from `@T_GetLeaf@12` return value through the terrain polygon builder to how the surface class indexes into `_tmapRemapTable`.* See [formats/T2.md](formats/T2.md).
